# CLAUDE.md

Інструкція для роботи з `vault/` — зовнішньою пам'яттю проєкту «Безпека Офісу — Суми».

## Навіщо vault

Контекстне вікно LLM — дефіцитний ресурс, а не сховище. ТЗ (`TZ_bezpeka_ofisu_sumy.md`) фіксує *що* будувати. Vault фіксує те, чого в ТЗ немає і що інакше довелось би відновлювати щосесії наново: чому обрано саме такі рішення, хід розробки, відкриті питання. Мета — щоб кожна нова сесія з Claude Code починалась не з нуля і не з перечитування всього репозиторію, а з точкового читання 2-3 файлів.

## Метод формування vault (за підходом Andrej Karpathy до context engineering)

Карпаті описує «вайб-кодинг» і роботу з LLM-агентами через принцип: **контекст — це те, що ти свідомо інженериш, а не те, що накопичується само**. Звідси конкретні правила для цього vault:

1. **Атомарність файлу.** Один файл — одна ідея/рішення/термін. Не «notes.md» на 500 рядків, а `decisions/0002-hybrid-llm-not-local.md` — файл, який можна прочитати цілком і зрозуміти без іншого контексту.

2. **Індекс — точка входу, не всі файли одразу.** `vault/INDEX.md` читається першим у кожній сесії. З нього переходити лише за посиланнями, які реально потрібні під поточну задачу — не «прочитати весь vault про всяк випадок». Це і є економія токенів: агент вантажить кіло­байти, а не весь vault.

3. **Розрізняти mutable і append-only пам'ять.**
   - `architecture/`, `glossary/` — стан «як є зараз», редагується in-place, стара інформація замінюється новою.
   - `decisions/`, `progress/` — історія. Рішення (ADR) не переписується заднім числом — якщо рішення переглянуте, старий файл лишається зі статусом `superseded`, а поруч з'являється новий. Прогрес — один файл на дату, ніколи не редагується заднім числом.

4. **Frontmatter в кожному файлі.** `title`, `updated`, `status` (`active` / `accepted` / `superseded` / `log`), за потреби `source` — посилання на розділ ТЗ. Дозволяє агенту (і людині) з першого рядка зрозуміти актуальність файлу, не читаючи весь вміст.

5. **Джерело правди — одне.** Vault не дублює ТЗ. Якщо факт вже точно й повно описаний у ТЗ — у vault лишається лише посилання (`source:` у frontmatter), а не копія тексту. Дублювання — це і зайві токени, і ризик розбіжності.

6. **Чому, а не що.** Найцінніше в `decisions/` — секція «Чому» і «Альтернативи відхилено». Код і ТЗ вже кажуть *що* зроблено; тільки vault пам'ятає, чому обрано саме цей варіант і що відкинуто — це саме те, що LLM-агент інакше вигадає заново при кожному рефакторингу.

7. **Періодичний prune.** Застарілі backlog-пункти, закриті питання — не накопичувати. Закритий пункт переноситься в `progress/<дата>.md` одним рядком і видаляється з `backlog/`. Vault, що постійно росте без чищення, деградує в той самий «контекст-мотлох», якого метод покликаний уникати.

## Структура

```
vault/
  INDEX.md              — точка входу, читати першою
  architecture/          — поточний стан системи (мутабельно)
  decisions/              — ADR: одне рішення = один файл (append-only)
  glossary/               — точні визначення домену (мутабельно)
  progress/                — журнал сесій, один файл на дату (append-only)
  backlog/                — відкриті питання, TODO (мутабельно, з prune)
  design/                  — дизайн-система UI (мутабельно)
```

## Правило: дизайн-система

Будь-яка робота над UI/UX цього проєкту (адмін-панель, дашборд, будь-які майбутні екрани) використовує **виключно** дизайн-систему з [vault/design/design.md](vault/design/design.md) — кольори, типографіку, layout, компоненти, анімації, стратегію теми (`data-theme` + CSS custom properties). Не вигадувати альтернативну палітру/компоненти і не змішувати з іншими UI-кітами.

Виняток — стек реалізації: `design.md` описаний під React (Horyzont), а це рішення для OfficeSecurity не діє (ТЗ розділ 11 — без React для адмін-панелі). Дизайн-токени й специфікація компонентів переносяться буквально; секція навігації/контенту (§9 design.md) — ні, вона під домен Horyzont. Деталі й обґрунтування — [ADR-0006](vault/decisions/0006-shared-design-system-no-react.md).

## Робочий цикл для Claude Code

- **На початку сесії:** прочитати `vault/INDEX.md`, звідти — лише релевантні файли під задачу.
- **Під час роботи:** якщо приймається архітектурне рішення — новий файл у `decisions/`; якщо змінюється компонент — правка `architecture/`.
- **Наприкінці сесії:** один запис у `progress/<YYYY-MM-DD>.md` — що зроблено, що відкрито, що далі.

## Graphify

[Graphify](https://github.com/Graphify-Labs/graphify) підключено 2026-07-22 — мапить код у knowledge graph, дозволяє запити (`graphify query`/`path`/`explain`) замість grep по файлах, локальна AST-обробка (без LLM). Встановлено глобально (`pip install --user graphifyy`, не в `requirements.txt` проєкту — це інструмент асистента, не залежність застосунку); `graphify-out/` (сам граф) в `.gitignore` — build-артефакт, не джерело правди.

**Використання:** `graphify path "A" "B"` — найкоротший шлях залежностей; `graphify explain "X"` — сусіди вузла X у графі. Якщо в проєкті є кілька файлів з однаковою назвою (напр. два `lexicon.py` — `app/common/` і `app/admin/routers/`), запит може бути неоднозначним — уточнювати повним шляхом.

**Граф не оновлюється сам** — після суттєвих змін коду перебудувати: `graphify extract . --code-only` (детерміновано, без LLM/API-ключа), потім `graphify cluster-only . --no-label` (без LLM-найменування кластерів). Деталі рішення — [vault/backlog/open-questions.md](vault/backlog/open-questions.md).
