## Огляд поточного стану + знайдений і виправлений збій юзербота

Користувач попросив перевірити й проаналізувати стан справ: чи все зроблено на поточному етапі, чи є що доопрацювати перед продовженням.

**Знайдено під час перевірки: юзербот-процес не працював.** `Get-CimInstance Win32_Process` показав лише два `uvicorn`-процеси (адмін-панель), жодного `app.userbot.main`. У логах — мережевий обрив (`ConnectionError: Connection to Telegram failed 5 time(s)` після 5 невдалих спроб reconnect до `149.154.167.51:443`), Telethon вичерпав автоматичні спроби й процес завершився без жодної обгортки-рестарту. `events_log` не отримував нових рядків щонайменше кілька годин (перевірено: `resolved_by` за останні 3 години — 0 рядків). Не повʼязано з жодним кодовим змінами цієї сесії — чиста мережева перервність, яка збіглась із тим, що ніхто не перевіряв стан процесу.

Перезапустив юзербот напряму — переконектився миттєво (мережа вже відновилась), `events_log` одразу почав отримувати нові рядки (перевірено `max(detected_at)` — щойно). Активних каналів — 9, без змін.

**Підсумок огляду для користувача:**
- Рівень 1+2 конвеєра виявлення — реалізовані, перевірені на живих даних, стабільні
- Свідомо прийняті обмеження (PSA "тривог", успадкування на не пов'язану тему) — задокументовані раніше, не потребують дій зараз
- Три кандидати на подальшу роботу озвучені користувачу: (1) надійність юзербота — немає process-supervisor/авто-рестарту навіть локально, збій цієї сесії саме тому і стався непоміченим; (2) відкладена колізія районних топонімів з вулицями Сум (`toponyms.yaml`); (3) Етап 3 плану (видимість нерозв'язаного). Користувач: жодне не зараз, лишити огляд стану як є — новий пункт (1) додано в backlog для майбутньої сесії, решта два вже там були.

## Підключення Graphify

Користувач нагадав про давній TODO з `CLAUDE.md` — підключити [Graphify](https://github.com/Graphify-Labs/graphify) (knowledge graph коду замість grep-навігації), відкладений відколи репозиторій був майже порожнім.

**Перша перевірка через WebFetch сторінки репозиторію дала тривожні сигнали** — назва pip-пакета `graphifyy` (подвійне "y"), нетипово великі цифри (93.3k зірок, YC S26) для інструменту, про який раніше ніхто з нас не чув. Розцінив це як можливий тайпсквотинг і прямо сказав користувачу не встановлювати, перш ніж перевірю глибше.

**Повторна перевірка через WebSearch (незалежні джерела, не сторінка самого репозиторію)** — блог Augment Code, DEV Community, окремі особисті блоги — незалежно підтвердили той самий опис інструменту й пояснили подвійне "y": ім'я `graphify` вже зайняте на PyPI непов'язаним пакетом, тому офіційний пакет зареєстровано як `graphifyy`, CLI-команда лишилась `graphify`. Скасував попереднє застереження прямо перед користувачем, пояснивши чому.

**Встановлення:**
- `uv`/`pipx` не було в оточенні — встановив через системний `pip install --user graphifyy` (не в `.venv`/`requirements.txt` проєкту — інструмент асистента, не залежність застосунку). Виявилось, що пакет 0.9.15 уже стояв у системі раніше
- `graphify install --platform claude` — зареєстрував skill глобально (`C:\Users\Admin\.claude\`), проєктний `CLAUDE.md` не займав
- `graphify extract . --code-only` — побудував граф повністю локально (детермінований AST, без LLM/API-ключа): 175 вузлів, 349 звʼязків, 22 спільноти (39 code-файлів; `schema.sql` пропущено — відсутня опційна `tree_sitter_sql`, не критично)
- `graphify cluster-only . --no-label` — згенерував `GRAPH_REPORT.md`/`graph.html` без LLM-найменування кластерів (щоб не залежати від API-ключа на цьому кроці)
- Перевірив на реальних запитах: `graphify path "handlers.py" "channel_state.py"` і `graphify explain "lexicon"` — обидва коректно показали справжні залежності нашого коду

**Housekeeping:** `graphify-out/` (сам граф) додано в `.gitignore` — build-артефакт, легко перебудувати, не комітиться. Оновив TODO-секцію в `CLAUDE.md` на "Graphify" з інструкцією використання/перебудови для майбутніх сесій, і позначив пункт у backlog виконаним.
