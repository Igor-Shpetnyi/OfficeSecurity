## Реальний LLM — Anthropic → Gemini того ж дня

Перепитав через AskUserQuestion, чи це означає підтвердити бюджетне рішення ADR-0012 просто зараз (воно й досі не було підтверджено) — так, підтверджено. Провайдера користувач делегував мені; порадив Anthropic Claude Haiku (обґрунтування — короткі структуровані задачі, обидва дешеві tier впорались би однаково, обрав через власний досвід інтеграції).

**Plan Mode:** через складність (реальні гроші, новий SDK, graceful degradation так, щоб Рівні 1-2 не зупинялись, якщо LLM недоступний — ТЗ §13) увійшов у режим планування. Дослідив кодову базу (ADR-0002, `app/config.py`, `.env`/`.env.example`, ТЗ §13 буквально, поточну форму `llm.py`) через Explore-агента, спроектував архітектуру через Plan-агента, сам виправив дві речі з чорнового плану агента до фіналізації: (1) `anthropic.AsyncAnthropic`, не синхронний клієнт — синхронний виклик всередині `async def` заблокував би event loop юзербота на час запиту, затримуючи ВСІ наступні повідомлення (Telethon `sequential_updates=True`, ADR-0010); (2) tool-use для форсованого JSON замість непідтвердженого `output_config`-параметра, який агент запропонував — не покладатись на здогад про форму API.

**Реалізовано повністю:** `app/common/llm.py` переписаний — лінивий модульний клієнт (той самий принцип, що завантаження словників у `lexicon.py`), circuit breaker (ТЗ §13: 3 збої → пауза 30с), timeout 4с/`max_retries=0`, нова гілка `is_new_target` у `channel_state.resolve()` (LLM тепер реально може сказати "не пов'язана тема" — закриває задокументоване в ADR-0012 обмеження аеростатів). Потрібен `text_snippet` у слоті Рівня 2 для tie-break — додав, threading `text` через `resolve()`/`_write_explicit()`.

**Перевірено 21 ізольованим сценарієм** з підміненим клієнтом (реального ключа нема — не можу створити акаунт/оплатити за користувача): відсутність ключа, всі переходи circuit breaker, успіх, галюцинації (низька впевненість/індекс поза межами), наскрізно через `channel_state.resolve()`. Усі пройшли. Перезапустив процеси, 0 помилок, регресій нема.

## Пивот: "поки виключно безкоштовні рішення"

Одразу після — користувач: "Давай поки не будемо націлюватись на Anthropic Claude API, а будемо використовувати виключно поки безкоштовні рішення". Запропонував три варіанти (Google Gemini безкоштовний тариф / локальна модель Ollama / повернутись до стаба) через AskUserQuestion — обрано Gemini.

Помітив: це та сама умова, яку **ADR-0002 явно передбачила** ("revisit condition: only go local if budget becomes strictly $0"), лише спрацював проміжний варіант (безкоштовний хмарний tier), не local — той самий дух рішення (не витрачати dev-час на інфраструктуру заради економії кількох доларів), інший конкретний вибір.

**Заміна архітектурно тривіальна** — та сама структура (async, circuit breaker, fail-closed, дві ролі), інший SDK. Перевірив актуальну форму `google-genai` (WebSearch + WebFetch README `googleapis/python-genai`, той самий принцип обережності, що з Anthropic tool-use) — `genai.Client(...).aio.models.generate_content()`, `gemini-2.5-flash`, `response_json_schema` для форсованого JSON. Переписав `llm.py`, адаптував тестовий скрипт (22 сценарії, +1 новий на "сирий" не-JSON текст від моделі) — усі пройшли. Перезапустив процеси, 0 помилок.

**Оскільки заміна сталась ДО першого коміту** — жодної заплутаної git-історії "додали Anthropic, відкотили", просто фінальний стан: Google Gemini, `GEMINI_API_KEY`. Закомічено ([c406327](https://github.com/Igor-Shpetnyi/OfficeSecurity/commit/c406327)).

## Перший реальний ключ — і одразу дві живі знахідки

Користувач додав справжній `GEMINI_API_KEY`. Перезапустив процеси, перший живий виклик одразу впав на фолбек (`stub_reason=api_error`). Обидві причини знайшов прямим викликом сирого SDK в обхід власного широкого `except Exception` (щоб побачити реальний exception):

1. **Timeout 4с — API Gemini вимагає мінімум 10с** (`400 INVALID_ARGUMENT: Manually set deadline 4s is too short`). Підняв до 12с.
2. **`gemini-2.5-flash` (пінована назва з офіційних прикладів) — 404, недоступна для нових ключів.** `gemini-2.0-flash` теж — 429, нульова безкоштовна квота саме на цю пінован назву. Перевірив `client.aio.models.list()` напряму — `-latest`-аліаси працюють і мають реальну квоту. Переключив на `gemini-flash-lite-latest`.

**Перевірив обидві ролі напряму на реальному API після фіксу:** компонування тексту — природний, змістовний результат ("Увага! У Сумах зафіксовано рух БПЛА..."); tie-break — на генуїнно неоднозначному прикладі модель чесно повернула `confidence: "low"` і fail-closed коректно впав на евристику (правильна поведінка, не баг), на однозначному прикладі — впевнено обрала правильну ціль. Живий скріншот `/notifications` показав обидва режими одночасно (свіжа картка з `⚠️ без ШІ-верифікації` з перехідного вікна між фіксами, старіша — на шаблоні).

**Урок, вартий занотувати окремо:** учорашні 22 мок-тести бездоганно перевірили логіку НАВКОЛО API (circuit breaker, парсинг, фолбеки), але структурно не могли впіймати жодну з цих двох проблем — обидві специфічні для реального стану зовнішнього сервісу на конкретну дату. Не недолік мок-тестування, а межа того, що воно взагалі здатне перевірити без реального ключа.

## Повна доказова стрічка "на основі яких повідомлень"

Користувач попросив бачити в "Сповіщеннях" не лише лічильник підтверджень, а самі повідомлення, з яких склалось рішення. `source_event_log_id` цього не покривав — мовчазні підтвердження (той самий/нижчий рівень від нового каналу) навмисно не пишуть окремий рядок у `threat_notifications`, тож їхній внесок був невидимий у UI.

**Реалізовано:** нова колонка `threat_state.contributing_event_ids INT[]` — накопичується на КОЖНОМУ сигналі (нова ціль/ескалація/мовчазне підтвердження/status), не лише на тому, що спричинив запис переходу. `threat_notifications.contributing_event_ids` — знімок цього масиву на момент переходу (той самий принцип, що вже `confirmation_count`/`contributing_channels`). УІ — розгортний `<details>` "На основі яких повідомлень (N)" під кожною карткою, перевикористав `.event-history-list` з `/events` (та сама CSS, що історія редагувань/ланцюжки відповідей), а не новий компонент.

**Перевірено ізольовано** (3 синтетичні повідомлення → 1 ціль → ескалація): накопичення точне й у правильному порядку, знімок "нова ціль" містив лише перше повідомлення, знімок "ескалація" — усі три. Живий скріншот підтвердив розгортання з коректним хронологічним списком (час/канал/текст) трьох різних повідомлень. По дорозі підхопило й реальне живе повідомлення з "🤖 ШІ" (не шаблон) — конвеєр стабільний, 0 помилок за весь час перевірки.
