## Баг: колір ідентифікації каналу не відповідав аватарці (4 з 10 каналів)

Користувач помітив, що колір каналу "Повітряні Сили ЗС України" (`@kpszsu`) не відповідає ТЗ (дизайн-системі/логіці ADR-0009). Аватарка — темно-синій фон + жовтогарячий тризуб, збережений колір — червоний `#cb4d4d`.

**Причина, дві накладені проблеми (`app/common/avatars.py`):**
1. `img.resize((1,1))` без явного `resample`-фільтра використовував дефолтну бікубічну інтерполяцію — не справжнє усереднення пікселів, а інтерполяційний артефакт при зменшенні до 1×1
2. Навіть із коректним усередненням (`Image.Resampling.BOX`) цей конкретний логотип дає майже ахроматичний сірий піксель (`(76,72,72)`, `sat=0.027`) — контрастні насичені зони (синій фон + жовтогарячий символ) у сирому RGB-усередненні гасять одна одну. Hue з майже сірого пікселя чисельно нестабільний (шум у кілька одиниць RGB довільно перевертає значення), а `_legible()` все одно перетворює його на "впевнений" насичений колір — довільний і не пов'язаний з реальним зображенням

Перевірка всіх 10 завантажених аватарок показала: **4 з 10** каналів (`@sumygo`, `@kpszsu`, `@sumy_sumy1`, `@sumyliketop`) мали `sat < 0.15` у сирому середньому — тобто той самий баг, не поодинокий випадок.

**Виправлено:** `extract_avg_color()` тепер (а) використовує `Image.Resampling.BOX` для справжнього усереднення, (б) повертає `None`, якщо `sat < 0.15` — виклик тоді падає на вже наявний `fallback_color()` (детермінований hue з хеша назви), той самий механізм, що вже спрацьовував для каналів без фото взагалі. Скинув `avatar_color = NULL` для 4 постраждалих каналів напряму в БД, перезапустив обидва процеси — `@kpszsu` тепер `#8ecb4d` (стабільний процедурний зелений), не випадковий червоний.

Користувач запитав, чому фолбек — хеш назви, а не колір "з палітри" аватарки (напр. найнасиченіший піксель без повної квантизації). Пояснив: dominant-color-via-quantize вже свідомо відхилений в ADR-0009 через складність для 38×38px UI-акценту; фолбек лише перевикористовує наявний механізм. Запропонував легший проміжний варіант (найнасиченіший піксель) — **користувач відхилив, лишити як є**. Зафіксовано в ADR-0009, щоб не re-litigate без нового приводу.

## Позначка медіа замість голого "—" у стрічці подій

Користувач помітив рядки з текстом "—" у "Події". Перевірено напряму через Telegram API (`client.get_messages`) — усі виявились фото без підпису (`MessageMediaPhoto`, `text=''`), не помилкою: наша система свідомо не тягне медіа, лише текст, тож порожній `raw_text` для фото-поста — очікувано.

**Побічне питання від користувача:** чи можна підтягнути "пакет смайликів, який використовує сам Telegram", щоб емодзі виглядали як у ньому. Перевірено в браузері — звичайні Unicode-емодзі (✅🔴📞) вже коректно рендеряться, просто шрифтом ОС (Windows Segoe UI Emoji), не стилем Telegram. Пояснив нюанс: власні емодзі-асети Telegram-клієнтів не обов'язково вільно ліцензовані для передруку, є перевірена альтернатива (Twemoji, MIT). **Користувач вирішив залишити як є** — не re-litigate без нового приводу.

**Реалізовано:** нова колонка `events_log.media_label` (`app/db/schema.sql`) — людська позначка типу медіа ("📷 фото", "🎥 відео", "😀 стікер" тощо), обчислюється один раз при вставці в `app/userbot/handlers.py::_media_label()` за атрибутами Telethon-події (`.photo`/`.video`/`.gif`/`.sticker`/`.voice`/`.audio`/`.poll`/`.document`, порядок важливий — gif технічно теж document/video під капотом). `app/common/events_query.py` — `media_label` протягнуто через `_word_diff_html()`/`_parse_history()`/reply-chain query, скрізь де раніше був голий `escape("—")` фолбек для порожнього тексту. Оновлено 3 шаблони (`_events_content.html` — основний текст, reply-прев'ю, ланцюжок відповідей; `_dashboard_content.html` — компактна стрічка на дашборді).

**Бекфіл історичних рядків:** 13 уже існуючих рядків з порожнім `raw_text` — усі перевірені напряму через Telegram API (не просто припущені) як фото, проставлено `media_label='📷 фото'` напряму в БД.

## Підозра користувача: фото+текст показує лише "—" — розслідування й переробка на заглушки

Користувач: "якщо в телеграмі в повідомленні є фото і текст, то в панелі відображається лише -". Перевірено напряму через Telegram API — **підтвердилось, причина: альбоми**. Кілька фото одним постом з одним підписом — Telegram кладе підпис лише в ОДНЕ повідомлення `grouped_id`-групи, решта фото приходять з порожнім `raw_text` по-справжньому (не наш баг захоплення). У Telegram це один пост, у нас — N окремих карток, лише одна з текстом. Побічно виявлено: деякі повідомлення отримують медіа через `edit`, не одразу при створенні (`events_log` це коректно відображає різним `media_type` для 'new'/'edit' рядків одного повідомлення).

Користувач: "давай замість вкладених файлів робитимемо заглушки... зроби так, щоб у подальшому була можливість реалізувати підставляти самі оригінальні вкладення". Переробив учорашню (тогожденну) чернетку `media_label` (готовий рядок-з-емодзі в БД) на структуровану модель — деталі й обґрунтування в новововеденому [ADR-0011](../decisions/0011-media-stub-structured-type.md):

- `events_log.media_type` (короткий код: photo/video/gif/sticker/voice/audio/poll/document/other) + `grouped_id` (про запас під майбутнє об'єднання альбомів) замість `media_label`
- `app/common/media.py` — єдина мапа код→підпис; `app/admin/templates/_media_icon.html` — Jinja-макрос з SVG-іконкою на тип
- Заглушка (іконка + підпис у власному блоці, `.attachment-stub` CSS) показується **незалежно від наявності тексту**, не лише як фолбек на порожній рядок — саме це виправляє скаргу: пост "фото + підпис" тепер показує і заглушку, і повний текст одночасно
- **Знайдено і виправлено власною перевіркою** (звірка кількості `.attachment-stub` в HTML з прямим SQL того самого набору подій, не користувачем): дашборд спершу показував заглушку лише коли текст порожній (`{% elif %}`), а не поруч із текстом, як на основній сторінці подій — розсинхрон між двома шаблонами. Виправлено для консистентності
- Backlog: об'єднання альбомів в одну картку і реальні вкладення замість заглушок — свідомо не зроблено зараз, окремі майбутні кроки

## Дослідження реальних Telegram-повідомлень і перегляд плану виявлення загроз

Користувач попросив проаналізувати повідомлення напряму з Telegram-каналів (не з нашої БД) за тиждень і продумати, чи справді чистий RegEx-двигун з ТЗ п.15 дасть прийнятний результат — з явним дозволом відходити від буквального ТЗ.

Витягнув повну історію напряму через Telegram API (наша `events_log` мала 4-денний провал через даунтайм юзербота) — **9917 повідомлень за 7 днів, 9 каналів**. Кількісний аналіз (широкий чорновий словник тригерів, розмітка реклами/шуму, кластеризація по 10-хв вікнах для мультиканальної кореляції, fuzzy-кластеризація топонімів) показав: структуровані джерела (Повітряні Сили ЗС України — 94%) регекс ловить відмінно, але канали живого спостереження (RDS-prostir 26%, Женьок Вещає 54% коротких повідомлень) ведуть репортаж еліптично — топонім чи дієслово стану без жодного тригер-слова, і без Telegram `reply_to` (перевірено напряму: `reply=None`). Це структурна проблема, не прогалина словника.

**Рішення** (детально — [ADR-0012](../decisions/0012-three-layer-detection-not-regex-then-llm.md), повна специфікація — [`TZ_konveyer_analizu_zagroz.md`](../../TZ_konveyer_analizu_zagroz.md)): трирівневий конвеєр замість послідовного RegEx→LLM — (1) лексика+газетир, (2) стан активної цілі на канал (Redis, TTL 20хв), (3) LLM як вузький fallback/tie-break, не класифікатор першої лінії. Це і знімає ризик rate-limit LLM API саме в пікові моменти масованої атаки (ТЗ п.16) для самого факту виявлення.

Користувач попросив детальний план реалізації з видимими на кожному кроці змінами в адмін-панелі — увійшов у Plan Mode, дослідив кодову базу (схема БД вже мала `regex_matched_level` невикористаним відколи ТЗ §14; FastAPI router-патерн; `live-refresh.js`), написав і затвердив план на 4 етапи (1: лексика+газетир, 2: стан каналу, 3: видимість нерозв'язаного, 4: LLM — гейтовано окремим рішенням про бюджет).

## Аудит перших живих повідомлень Рівня 1

Користувач знайшов конкретний кейс: `"🛵 БпЛА на Суми з півночі"` отримало геомітку, але не рівень загрози — попросив проаналізувати всі повідомлення з моменту підключення патчу. Витягнув усі 36 рядків (`detected_at >= час першого обробленого рядка`) і перевірив кожен вручну проти сирого тексту.

**Знайдено й виправлено:**
1. `бпла` без уточнення типу не було в словнику взагалі (лише конкретні типи чи `невідом`) — додано в `orange` (ТЗ §6: невідомі БпЛА трактуються як потенційно ударні)
2. `🟢 Сумська область - відбій повітряної тривоги!` показувало `🔴 red` поруч зі status-бейджем "відбій" — `тривог`-стем ловить слово незалежно від контексту. Додав `lexicon.resolve_level()` — гасить рівень, якщо в повідомленні є status-маркер
3. `🔴Тревога` (російський правопис) не зловилось україномовним стемом — додано `тревог`

Backfilив вже збережені рядки під виправлену логіку, перезапустив юзербот. Верифікував фікси ізольованими тестами (включно з genuine-кейсом "Оголошена повітряна тривога, всім в укриття" — має лишитись red, і лишився).

**Знайдено, свідомо НЕ виправлено:** два повідомлення (пояснювальний пост про систему оповіщення "Вуличне радіо"; стара новина з фразою "не ігноруйте сигнали тривоги") отримали `🔴 red` за згадку слова "тривога" в описовому, не декларативному контексті, без status-маркера поруч (тож фікс п.2 не покриває). Точкові евристики під конкретні фрази відхилені свідомо — нескінченна гонитва замість розв'язання класу проблеми. Задокументовано детально в [ADR-0012 ("Уточнення 2026-07-21")](../decisions/0012-three-layer-detection-not-regex-then-llm.md) і розділі 7 [`TZ_konveyer_analizu_zagroz.md`](../../TZ_konveyer_analizu_zagroz.md) як конкретний, не гіпотетичний, вхідний матеріал для проєктування Рівня 3 (LLM).

## Звірка рівнів словника з таблицею часу реакції ТЗ §6

Користувач зауважив: `каб`/`ракет`/`рсзв` стояли в `orange`, хоча за ТЗ §6 їхній час реакції (КАБ 2-5 хв, артилерія/РСЗВ 0-15 сек) — найкоротший у всій таблиці, замалий для проміжного рівня "спостерігати". Систематично звірив КОЖНЕ orange-слово з таблицею ТЗ §6 (не лише 3 названі), результат:

- **→ red** (короткий час реакції, 0-5 хв): `каб`, `ракет`, `рсзв`, `реактивн`, `швидкісн` (ТЗ §6: "Високошвидкісні (реактивні) ~5 хв підльоту"), і сленгове `уєбало`/`в'єбеніло` (підтверджене влучання — той самий сенс, що вже-red `вибух`)
- **лишились orange**: `шахед`/`герань` (20 хв - 1 год за ТЗ §6 — саме цей запас часу виправдовує "стежити", не "негайно в укриття"), `фпв`/`ланцет`/`молні`/`наліт`/`невідом`/`бпла`
- **→ yellow** (сигнали-провісники, не підтверджена загроза — той самий клас, що розвідувальні БпЛА): `авіаці`/`т/а` (активність тактичної авіації передує пуску), `дроч`/`болванк` (сленг невизначеної цілі, часто фейк за реальними даними)

Переписав `triggers.yaml` з поясненням логіки розподілу (час реакції, не "наскільки страшне слово") у заголовку файлу. Перевірив ізольовано на прикладах, потім **перепрогнав Рівень 1 по всій наявній історії** (635 рядків `events_log`, не лише нові повідомлення) — написав одноразовий скрипт, що перерахував `regex_matched_level`/`matched_status`/`matched_location`/`resolved_by` для кожного рядка з поточним лексиконом. Розподіл після перерахунку: 185 red, 105 orange, 17 yellow, 329 без рівня. Перезапустив юзербот під оновлений словник.

## Трасування рішень конвеєра — "чому", не лише "що" ([ADR-0013](../decisions/0013-decision-trace-every-layer.md))

Користувач попросив закласти наскрізний принцип: по всьому конвеєру, для кожного повідомлення (чи стеку повідомлень), має десь фіксуватись ЧОМУ й ЧЕРЕЗ ЩО ухвалено те чи інше рішення — щоб аналізувати не доводилось перечитуванням коду й ізольованими тестами (саме так робився вчорашній аудит).

Реалізував для Рівня 1 (єдиного живого зараз), заклавши форму, яку Рівень 2/3 розширюватимуть, а не переви­нахо­дитимуть:
- `app/common/lexicon.py::DecisionTrace` + `analyze()` — нова єдина точка входу, рахує рівень/статус/локацію ОДНИМ проходом і одразу повертає evidence людською мовою ("зловлено слово «каб» (red)", "рівень «red» приглушено — знайдено status «відбій»", "fuzzy-збіг 0.91 — «вакплівщина» проти форми «вакалівщина»"). `handlers.py` тепер викликає лише `analyze()` замість трьох окремих `match_*()` — значення й пояснення рахуються з тих самих даних одним викликом, не можуть розійтись
- `events_log.decision_trace JSONB` — нова колонка, `{"layer": "lexicon", "level_evidence": ..., "status_evidence": ..., "location_evidence": ...}`
- UI: розгортний блок "Чому?" (`decision_explanation()` в `_level_badge.html`, той самий патерн `<details>`, що історія редагувань) під бейджами рівня в `/events`
- Перепрогнав `decision_trace` по всій наявній історії (656 рядків, той самий backfill-підхід, що для реклассифікації рівнів) — і нові, і старі повідомлення мають пояснення в панелі
- Перевірено на живому потоці напряму в браузері — блок "Чому?" коректно показує `"зловлено слово \"бпла\" (orange)"`, `"зловлено слово \"відбій\""` тощо

## Реалізація Етапу 2 — стан активної цілі на канал

Наступний етап затвердженого плану ([ADR-0012](../decisions/0012-three-layer-detection-not-regex-then-llm.md)): успадкування рівня для повідомлень без тригер-слова, поки в каналу є свіжа активна ціль.

**Код:**
- `app/common/channel_state.py::resolve()` — приймає `DecisionTrace` від `lexicon.analyze()`, повертає новий (чи той самий) `DecisionTrace`. Redis-модель `channel_state:{channel_id}:{slot}` (0-2 слоти), TTL 1200с (20 хв, та сама константа "локаційно втраченої цілі"). Вибір слота при явному сигналі: спершу той, що з тою самою локацією, інакше вільний, інакше витісняється найстаріший
- `handlers.py`/`main.py` — прокинуто `redis_client` до `_store_event`/`register_message_handler`, замінено виклик голого `lexicon.analyze()` на `channel_state.resolve(...)`
- UI: нова сторінка `/channels/state` (`channel_state.py` роутер + `channel_state.html`/`_channel_state_content.html`) — жива таблиця активних слотів, live-refresh 5с. Бейджі подій (`origin_map` в `_level_badge.html`) розрізняють `🔗 успадковано`/`❓ неоднозначно` — закладено заздалегідь ще на Рівні 1, тепер справді використовується

**Знайдено й виправлено власним ізольованим тестом до підключення в handlers.py:** повідомлення-status ("Відбій загрози балістики") успадковувало рівень активного слоту замість того, щоб закрити його — `resolve()` перевіряв лише `lex_trace.level`, не `lex_trace.status`. Виправлено `_clear_matching_slot()` — status-сигнал явно закриває відповідний слот.

**Перевірено на живому потоці:** дочекався кількох реальних повідомлень — `resolved_by='channel_state'` з'явився в `events_log` природно (канал "Повітряні Сили" написав про аеростати для БпЛА, зловило "бпла"; за кілька хвилин активність каналу успадкувала orange). `/channels/state` показує живий слот з коректним TTL, що спадає.

**Знайдено на живих даних, свідомо НЕ виправлено (рішення користувача):** та сама історія з аеростатами за ~9 хв дала й хибне спрацювання — канал опублікував зовсім не пов'язану статтю про соцдопомогу (+2 фото), і вона теж успадкувала orange, бо в ній просто немає тригер-слова, а модель "1 активний слот = продовження" не розрізняє зміну теми. Користувач: лишити як є, задокументувати як відоме обмеження, не патчити евристикою — кандидат для Рівня 3 (LLM). Задокументовано детально в ADR-0012 і backlog.

## UX-правки `/channels/state` за проханням користувача

- **TTL мав тікати щосекунди, не раз на 5с (цикл live-refresh).** Додав `startTtlTicker()` в `live-refresh.js` — сервер віддає `ttl_expires_at_ms` (epoch), клієнт інтерполює відлік щосекунди між fetch-циклами, читаючи DOM наново (автоматично підхоплює елементи після заміни innerHTML)
- **"Джерело" (`#172662`) не давало зрозуміти, про яку подію йдеться.** Додав `id="event-{channel}-{msg_id}"` на кожну картку в `/events` і посилання `/events/recent#event-...` з таблиці активних цілей
- **Власна перевірка виявила побічний баг:** нативний CSS `:target` для підсвітки картки губився після першого ж циклу live-refresh (елемент з тим самим id з'являється заново після заміни `innerHTML`, але браузер уже не рахує його "тим самим target"). Виправив — підсвітка тепер JS-клас (`hash-highlight`), що `live-refresh.js` переприкладає після кожного оновлення, той самий патерн, що вже є для збереження відкритих `<details>`. Перевірено: підсвітка з'являється одразу і переживає цикл рефрешу
- **Користувач попросив довиправити:** картка мала центруватись по вертикалі при переході (не просто прогортатись до верху), і жовта обводка мала зникати сама через кілька секунд, а не лишатись назавжди. Переробив `highlightHashTarget()` на одноразову дію за завантаження сторінки: `scrollIntoView({block: 'center'})` для центрування, `setTimeout` знімає клас через 3с, і одразу після підсвітки `history.replaceState` чистить hash з URL — щоб наступні цикли live-refresh (кожні 5с) не намагались повторити те саме знову. Додав CSS `transition` на `border-color`/`box-shadow`, щоб зникнення було плавним, не миттєвим. Перевірено: центр картки збігається з центром в'юпорта пікселем-у-піксель, підсвітка знімається автоматично

**Забутий перезапуск адмін-панелі:** користувач помітив, що `/lexicon` досі показує старий розподіл рівнів. Причина — `lru_cache` в `app/common/lexicon.py` кешує YAML окремо в КОЖНОМУ процесі; перезапустив тоді лише юзербот, адмін-панель (окремий процес) тримала словник, завантажений ще до правки. Перезапустив і адмін-панель, перевірив `/lexicon` напряму в браузері — розподіл коректний (16 red / 9 orange / 7 yellow). Задокументовано як стандартну готчу в `architecture/overview.md`, щоб не наступати вдруге.

## Одна локація на повідомлення — структурна втрата інформації

Користувач знайшов кейс: `"🛵 Сумщина: групи БпЛА ➡️ курсом на Шостку, Ямпіль, Глухів."` показувало бейдж лише `📍 Глухів`, хоча повідомлення називає маршрут через кілька топонімів. Розслідував — причина подвійна: `Шостка`/`Ямпіль` відсутні в газетирі (додав), і `match_location()` структурно повертав лише одну (найдовшу пряму) знайдену локацію, відкидаючи решту — навіть з повним газетиром втрата лишилась би.

Запитав користувача: повна підтримка кількох локацій, чи лише доповнити газетир і лишити логіку "одна локація". Обрано перше. Реалізував наскрізно:
- `lexicon.py`: `DecisionTrace.location`/`location_evidence` — кортежі; `_match_locations_explained()` перебирає весь газетир, не зупиняється на першому збігу
- `events_log.matched_location VARCHAR(100)` → `matched_locations TEXT[]` (перейменовано, стару колонку дропнув — семантика змінилась)
- `channel_state.py`: слоти зберігають список локацій, вибір/закриття слота — за перетином множин, не рівністю
- `_level_badge.html`: цикл замість одного `if` — кожна локація власним бейджем і рядком "Чому?"

**Наступив на власну помилку під час рестарту:** забув перебекфілити старі рядки після `ALTER TABLE` (додав нову колонку, стару дропнув) — старі рядки лишились з `NULL`, шаблон впав `TypeError: 'NoneType' object is not iterable` на першому ж GET `/events/recent/fragment` (побачив у логах адмін-панелі одразу після рестарту). Виправив захисно в шаблоні (`e.matched_locations or []`) і переобчислив `matched_locations`/decision_trace по всій історії (862 рядки) тим самим бекфіл-скриптом, що й раніше цієї сесії.

**Перевірено на живому потоці:** Basic Auth у браузерному інструменті завис на URL з embedded-credentials (та сама відома пастка з попередньої сесії) — перевірив натомість прямим HTTP-запитом (`curl -u`) до `/events/recent/fragment`. Знайшов реальне повідомлення "🛵 БпЛА на Шостку на Сумщині з півночі" з обома бейджами `📍 Суми` й `📍 Шостка` одночасно, і відповідні два пункти в "Чому?". **87 з 862 історичних рядків мають більше однієї локації** — це регулярна риса повідомлень, не рідкісний край.

**Побічно виявлено, не помилка:** форма `"сум"` у газетирі (для "Суми") — короткий підрядок, збігається й усередині "Сумщина". Раніше ховалось (перемагала довша форма іншого топоніма), тепер видиме як окремий бейдж `📍 Суми` на будь-якому повідомленні про Сумщину загалом. Не хибне по суті, але варта уваги при подальшій звірці газетиру.
