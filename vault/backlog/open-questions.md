# Backlog

## Майбутні інтеграції

- [ ] **Graphify** (https://github.com/Graphify-Labs/graphify) — підключити для якості вайб-кодингу та економії токенів. Мапить проєкт (код, документи) у knowledge graph, локальна обробка коду через tree-sitter AST (без LLM-викликів на цьому кроці), запити (`graphify query`, `graphify path`, `graphify explain`) замість grep по файлах. Кандидат на заміну частини ручної навігації по vault, коли той розростеться — наразі vault малий, ручної структури INDEX.md достатньо. Переоцінити після Етапу 1-2 розробки (див. ТЗ п.15), коли з'явиться реальний код для побудови графу.

- [x] **Docker для Postgres/Redis (локальна розробка)** — вирішено (2026-07-16): `docker-compose.yml`, застосунок лишається на systemd/venv. Див. [ADR-0005](../decisions/0005-docker-compose-local-infra.md).
- [ ] **Контейнеризація застосунку і/або деплой на проді (Hetzner)** — свідомо відкладено. Чи контейнеризувати юзербот/admin panel, чи ставити Postgres/Redis тим самим Docker Compose на VPS чи нативно — окреме рішення, буде розглянуто разом з усіма питаннями деплою пізніше.

## Відкриті питання з ТЗ

- [ ] Етап 2 (GeoJSON + Гаверсинус) — не починати, поки MVP на статичних списках не пропрацював мінімум місяць у бою (ТЗ п.8)
- [x] Автентифікація адмін-пульта на етапі розробки — тимчасово Basic Auth (`app/common/auth.py`), без VPN/whitelist. Остаточний вибір (Telegram Login Widget чи Basic Auth+VPN) — за фактом того, скільки адмінів і чи в них статичні IP, лишається відкритим для проду (ТЗ п.11)
- [ ] `channel_identifier` як UNIQUE — не дедуплікує один і той самий канал, доданий двічі різними ідентифікаторами (раз через @username, раз через invite-лінк). Не критично для MVP, але варто мати на увазі при рості списку каналів
- [x] `deactivate_channel` тепер реально виходить з каналу — вирішено (2026-07-16): `sync_pending_leaves()` + `LeaveChannelRequest`, статус `join_status='left'`. Протестовано наскрізно на `@telegram`
- [x] Порядок рядків у `events_log` не гарантовано хронологічний через конкурентні asyncio-хендлери — вирішено (2026-07-19): `sequential_updates=True`. Див. [ADR-0010](../decisions/0010-sequential-updates-race-fix.md)

## Відкрито за сесію 2026-07-19 (адмін-панель: канали/події)

- [ ] **Reply-прев'ю для повідомлення поза базою** — якщо канал приєднано пізніше за пост, на який відповідають, зараз просто заглушка "У відповідь на повідомлення поза базою (#ID)". Можна підвантажувати такий текст напряму з Telegram API за запитом (`client.get_messages`) — не зроблено, бо додає затримку/виклик на кожен такий випадок; поточна заглушка покриває більшість випадків
- [ ] **Клік по reply-прев'ю → прокрутка до батьківського повідомлення**, якщо воно є у видимій стрічці — дрібне UX-покращення, не реалізовано
- [x] **Кореляція reply-ланцюжків для RegEx/LLM-шару** — уточнено (сесія 07-21): пряме дослідження показало, що канали живого спостереження (RDS-prostir) взагалі НЕ використовують Telegram `reply_to` для власних продовжень (`reply=None`), тож саме по reply-графу кореляцію не збудувати. Замінено на per-канал стан активної цілі в Redis (TTL 20 хв). Див. [ADR-0012](../decisions/0012-three-layer-detection-not-regex-then-llm.md) і [TZ_konveyer_analizu_zagroz.md](../../TZ_konveyer_analizu_zagroz.md)

## Відкрито за сесію 2026-07-21

- [ ] **Реалізація Рівня 1+2 конвеєра виявлення (лексика+газетир, стан каналу в Redis)** — план і специфікація готові ([ADR-0012](../decisions/0012-three-layer-detection-not-regex-then-llm.md), [TZ_konveyer_analizu_zagroz.md](../../TZ_konveyer_analizu_zagroz.md)), сама реалізація — наступний крок
- [ ] **Словник тригерів і газетир топонімів — чернетка потребує звірки користувачем** — зібрано з одного тижня реальних даних (не з оригінального розділу 5 ТЗ, якого немає в репозиторії), перед використанням у продакшн-конвеєрі варто переглянути й доповнити
- [ ] **Об'єднання альбомів (grouped_id) в одну картку** — зараз кожне фото альбому (кілька фото одним постом з одним підписом) — окрема картка в стрічці, лише одна з них має текст (Telegram так зберігає, не наш баг — див. [ADR-0011](../decisions/0011-media-stub-structured-type.md)). `events_log.grouped_id` уже зберігається про запас, саме об'єднання (яка картка "головна", як показати "+N фото") не реалізовано — окрема фіча, чекає окремого запиту
- [ ] **Реальні вкладення замість заглушок** — `media_type` навмисно структурований (не готовий текст), щоб пізніше підставити завантажені оригінали (той самий підхід, що вже є для аватарок каналів, `app/common/avatars.py`) без переробки схеми/шаблонів. Не реалізовано — користувач явно попросив лише заглушки на цьому кроці
