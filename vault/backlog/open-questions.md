# Backlog

## Майбутні інтеграції

- [ ] **Graphify** (https://github.com/Graphify-Labs/graphify) — підключити для якості вайб-кодингу та економії токенів. Мапить проєкт (код, документи) у knowledge graph, локальна обробка коду через tree-sitter AST (без LLM-викликів на цьому кроці), запити (`graphify query`, `graphify path`, `graphify explain`) замість grep по файлах. Кандидат на заміну частини ручної навігації по vault, коли той розростеться — наразі vault малий, ручної структури INDEX.md достатньо. Переоцінити після Етапу 1-2 розробки (див. ТЗ п.15), коли з'явиться реальний код для побудови графу.

- [x] **Docker для Postgres/Redis (локальна розробка)** — вирішено (2026-07-16): `docker-compose.yml`, застосунок лишається на systemd/venv. Див. [ADR-0005](../decisions/0005-docker-compose-local-infra.md).
- [ ] **Контейнеризація застосунку і/або деплой на проді (Hetzner)** — свідомо відкладено. Чи контейнеризувати юзербот/admin panel, чи ставити Postgres/Redis тим самим Docker Compose на VPS чи нативно — окреме рішення, буде розглянуто разом з усіма питаннями деплою пізніше.

## Відкриті питання з ТЗ

- [ ] Етап 2 (GeoJSON + Гаверсинус) — не починати, поки MVP на статичних списках не пропрацював мінімум місяць у бою (ТЗ п.8)
- [x] Автентифікація адмін-пульта на етапі розробки — тимчасово Basic Auth (`app/common/auth.py`), без VPN/whitelist. Остаточний вибір (Telegram Login Widget чи Basic Auth+VPN) — за фактом того, скільки адмінів і чи в них статичні IP, лишається відкритим для проду (ТЗ п.11)
- [ ] `channel_identifier` як UNIQUE — не дедуплікує один і той самий канал, доданий двічі різними ідентифікаторами (раз через @username, раз через invite-лінк). Не критично для MVP, але варто мати на увазі при рості списку каналів
- [x] `deactivate_channel` тепер реально виходить з каналу — вирішено (2026-07-16): `sync_pending_leaves()` + `LeaveChannelRequest`, статус `join_status='left'`. Протестовано наскрізно на `@telegram`
- [ ] Порядок рядків у `events_log` не гарантовано хронологічний — конкурентні асинхронні обробники `NewMessage`/`MessageEdited` можуть записати `edit` раніше за `new` того ж повідомлення (спостережено 2026-07-16, різниця <2мс). Для RegEx/LLM-шару (наступні етапи) — корелювати події за `telegram_message_id`, не за порядком вставки/`id`
