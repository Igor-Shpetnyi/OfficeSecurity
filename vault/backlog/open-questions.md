# Backlog

- [x] **Graphify** — підключено 2026-07-22. Перед встановленням перевірив легітимність через незалежні джерела (не лише сторінку самого репозиторію) — пакет `graphifyy` (подвійне "y") виявився не тайпсквотом, а зареєстрованим під іншим ім'ям через колізію на PyPI з непов'язаним пакетом `graphify`, підтверджено кількома незалежними блогами/оглядами. Встановлено глобально (`pip install --user graphifyy`), skill зареєстровано для Claude Code (`graphify install --platform claude` → `C:\Users\Admin\.claude\` — не чіпає проєктний `CLAUDE.md`). Перевірено на реальних запитах (`graphify path`/`explain`) — коректно показує залежності `app/common/lexicon.py` тощо. `graphify-out/` в `.gitignore` — build-артефакт, перебудовується командою, не комітиться. Використання — див. `CLAUDE.md`

- [x] **Реалізація Рівня 1+2 конвеєра виявлення (лексика+газетир, стан каналу в Redis)** — реалізовано й підключено 2026-07-21. Див. [ADR-0012](../decisions/0012-three-layer-detection-not-regex-then-llm.md) ("Уточнення" від 21.07 для обох рівнів) і `vault/architecture/overview.md`
- [ ] **Успадкування Рівня 2 на тематично не пов'язані повідомлення** — знайдено на живих даних 2026-07-21: канал написав про БпЛА (orange), за ~9 хв опублікував зовсім не пов'язану статтю (+фото) без жодного тригер-слова — вона теж успадкувала orange, бо "1 активний слот = продовження" не розрізняє зміну теми. Користувач підтвердив: лишити як є, не латати евристикою — кандидат для Рівня 3 (LLM). Див. ADR-0012, "Уточнення 2026-07-21: реалізація й перший живий запуск Рівня 2"
- [ ] **"тривог" ловить будь-яку згадку слова, не лише живе оголошення** — аудит 2026-07-21 після першого живого запуску Рівня 1 знайшов 2 реальні хибні спрацювання: PSA-текст про роботу системи оповіщення й нагадування "не ігноруйте сигнали тривоги" в старому пості — обидва отримали 🔴 red, хоча не описують активну загрозу. Пара з status="відбій" вже гаситься (`lexicon.resolve_level`), але це не покриває PSA/пояснювальний текст без status-слова. Свідомо не латати спеціальними правилами під конкретні фрази — точкові евристики тут дадуть нескінченну гонитву; це саме той клас неоднозначності, що чекає на Рівень 3 (LLM). Див. коментар у `triggers.yaml`
- [ ] **Колізія районних топонімів з вулицями Сум** — прикметникові форми районних центрів (`білопільська`, `охтирська`, `хотінська` тощо) буквально збігаються з реальними назвами вулиць у самих Сумах (підтверджено офіційним реєстром Сумської міськради — `app/common/data/streets_sumy.yaml`, 648 записів). `Хотінь` вже має пряму колізію (`хотінська` — і форма топоніма, і назва вулиці), `Тополя`/`Баранівка`/`Битиця` — ризик через fuzzy-match. Користувач свідомо відклав правку 2026-07-21 ("занотуємо, повернемось пізніше") — не re-litigate без нового приводу, просто виконати, коли попросить
- [ ] **Підключити `streets_sumy.yaml` до matching-логіки** — список зібрано (2026-07-21, джерело: офіційний реєстр smr.gov.ua), але навмисно НЕ підключено до `app/common/lexicon.py::match_location()` — потребує окремої логіки (матчити лише в парі з "вул."/"вулиця" перед назвою, не голим підрядком), інакше масові хибні спрацювання на звичайних словах-назвах вулиць
- [ ] **Словник тригерів і газетир топонімів — чернетка потребує звірки користувачем** — зібрано з одного тижня реальних даних (не з оригінального розділу 5 ТЗ, якого немає в репозиторії), перед використанням у продакшн-конвеєрі варто переглянути й доповнити. Перший конкретний прохід зроблено 2026-07-22 (ручний аналіз усього кошика "нерозв'язано", 5 нових стемів додано — `балістичн`/`влуп`/`впердол`/`оптика`/`уламк`), але це не заміняє повного перегляду користувачем, лише закрило те, що спливло з нерозв'язаних
- [ ] **Об'єднання альбомів (grouped_id) в одну картку** — зараз кожне фото альбому (кілька фото одним постом з одним підписом) — окрема картка в стрічці, лише одна з них має текст (Telegram так зберігає, не наш баг — див. [ADR-0011](../decisions/0011-media-stub-structured-type.md)). `events_log.grouped_id` уже зберігається про запас, саме об'єднання (яка картка "головна", як показати "+N фото") не реалізовано — окрема фіча, чекає окремого запиту
- [ ] **Реальні вкладення замість заглушок** — `media_type` навмисно структурований (не готовий текст), щоб пізніше підставити завантажені оригінали (той самий підхід, що вже є для аватарок каналів, `app/common/avatars.py`) без переробки схеми/шаблонів. Не реалізовано — користувач явно попросив лише заглушки на цьому кроці

## Відкрито за сесію 2026-07-22

- [ ] **Юзербот немає process-supervisor/авто-рестарту навіть локально** — знайдено під час планового огляду стану 2026-07-22: процес не працював, `events_log` не отримував нових рядків кілька годин непоміченим. **Уточнення від користувача (2026-07-22):** причина — ручна зупинка процесу під час паузи в розробці (не сам собою мережевий збій під час активної роботи), тож це не підтверджений прод-подібний інцидент і не критично зараз. ТЗ §12 "Відмовостійкість юзербота" стосується прод-стійкості (Етап 5) — свідомо відкладено користувачем до цього етапу, не пріоритет до релізу
- [x] **Етап 3 плану (видимість нерозв'язаного, `/events?filter=unresolved`)** — реалізовано 2026-07-22, обрано пріоритетом після відхилення process-supervisor. 236 нерозв'язаних повідомлень на момент реалізації (переважно реклама/побутові пости без тригер-слів) — вхід для майбутнього рішення про бюджет Етапу 4 (LLM). Див. `vault/architecture/overview.md`
- [ ] **"Нерозв'язано" (Етап 3) не ловить "є локація, нема рівня"** — знайдено 2026-07-22 при фіксі "італмас": `resolved_by` стає не-NULL, щойно знайдено БУДЬ-ЩО (рівень, статус АБО локація) — тобто повідомлення з голою локацією без рівня вважається "розв'язаним" фільтром `/events?filter=unresolved`, хоча найважливіше поле (рівень загрози) все ще відсутнє. На практиці такі рядки все одно видно в бейджах (📍 без кольору), просто вони не потрапляють у лічильник "нерозв'язано" — для точнішого виміру варто розділити умову на "нема рівня" окремо від "нічого не знайдено", якщо це стане важливим для майбутнього рішення про Етап 4
