# ADR-0012: Трирівневий конвеєр виявлення загроз

## Контекст

ТЗ п.15 передбачає Етап 1 — чистий RegEx-двигун (без LLM), Етап 2 — LLM-гібрид поверх нього. Перед стартом Етапу 1 користувач попросив дослідити, чи справді чистий RegEx дасть прийнятний результат, враховуючи реальну варіативність повідомлень — з явним дозволом відходити від буквального тексту ТЗ, якщо дані покажуть інше.

**Дослідження:** витягнуто повну історію напряму з Telegram API (не з нашої `events_log` — там був 4-денний провал через даунтайм юзербота) за 7 днів по всіх 9 активних каналах моніторингу — **9917 повідомлень**. Кількісний аналіз: широкий чорновий словник тригерів (~35 форм з реальних даних), розмітка реклами/шуму, кластеризація по 10-хвилинних вікнах для мультиканальної кореляції, fuzzy-кластеризація топонімів (`difflib`, поріг схожості 0.82), і головне — виокремлення коротких повідомлень, які не зловив жоден тригер.

## Виявлені факти

- **Структуровані джерела — regex ідеальний.** «Повітряні Сили ЗС України»: 999/1060 (94%) повідомлень зловлено словником — суворо шаблонний формат (`🛵`/`🚀`/`☄️`/`📢` + фіксовані фрази), по суті вже готовий enum.
- **Канали живого спостереження — структурний провал регексу.** RDS-prostir: 228/890 (26%) коротких повідомлень і «Женьок Вещає»: 610/1126 (54%) — еліптичні продовження без жодного тригер-слова: голий топонім («Баранівка») чи дієслово стану («Впала», «Далі Х», «Повернув на Х»). Сленгові індикатори прильоту без словникового аналога: «уєбало», «в'єбеніло».
- **Підтверджено напряму, що це не проблема reply-графа:** еліптичні повідомлення в Telegram НЕ мають `reply_to_message_id` (перевірено на конкретному прикладі: `id=47223 reply=None` — «Далі Старе Село, Новоселиця»). Покладатись на `reply_to` (як передбачав backlog-пункт з сесії 07-19) недостатньо — треба хронологічна память по каналу, не граф відповідей.
- **Відмінки — окрема, механічно розв'язна проблема.** 38 fuzzy-кластерів словоформ одного топоніма (напр. `Харківщини/Харківщину/Харківщина/Харківщині`). Статичний список з ТЗ §8 без нормалізації пропустить більшість форм.
- **Підтримка самого словника — нескінченна гонитва.** Власний чорновий тригер «фіксац» не зловив «фіксується»/«не фіксується» (інша словоформа) — конкретний приклад того, що розширення словника вручну ніколи не наздожене реальну мову каналів.

Повна методологія, цифри та приклади — [`TZ_konveyer_analizu_zagroz.md`](../../TZ_konveyer_analizu_zagroz.md).

## Рішення

Замість послідовності «Етап 1 = чистий RegEx → Етап 2 = LLM поверх», конвеєр виявлення будується як три рівні, де **LLM не стоїть на гарячому шляху первинного виявлення**:

1. **Лексика + газетир** (regex + fuzzy-match топонімів) — швидкий, безкоштовний, детермінований фільтр. Ловить структуровані джерела й явні «топонім+тип» повідомлення.
2. **Стан активної цілі на канал** (Redis, ключ `channel_state:{telegram_id}`, TTL 20 хв — перевикористання вже узгодженої константи «локаційно втраченої цілі» з ТЗ §10, а не нова довільна цифра). Повідомлення без тригера, що приходить, поки в каналу є свіжий активний запис, трактується як продовження цієї ж події (успадковує `threat_type`, оновлює `location`).
3. **LLM — три вузькі, не-критичні-по-часу ролі**, а не класифікатор першої лінії:
   - tie-breaker, коли в каналі кілька одночасних активних цілей і незрозуміло, до якої належить нове повідомлення (короткий структурований запит, не відкрита класифікація);
   - fallback-класифікація того, що не зловили рівні 1-2 (справді нова лексика);
   - періодичний офлайн-прохід по «шумовому» кошику з пропозицією нових слів для словника — перетворює підтримку словника з ручного стеження на автоматизований процес.

## Чому

Еліптичність (повідомлення без ключових слів, сенс яких — у попередньому повідомленні того ж каналу) — це структурна властивість того, як конкретно ці канали ведуть репортаж, а не прогалина словника. Жодне розширення списку тригер-слів її не закриє, бо в самому реченні («Впала», «Баранівка») немає інформації, яку можна було б зловити — вона є лише в контексті. Це вимагає окремого механізму памʼяті, а не «кращого» regex.

Винесення LLM з гарячого шляху виявлення також знімає ризик з таблиці ТЗ п.16 (rate limit API саме під час масованої атаки, коли трафік каналів найбільший) — для самого факту виявлення загрози, бо рівні 1-2 обробляють переважну більшість реального сигналу незалежно від доступності зовнішнього API.

## Обмеження, визнані свідомо

Модель «один активний запис на канал» ламається, коли той самий канал одночасно веде дві незалежні цілі впереміш (реальний приклад у даних: «Кровне > Мар'ївка молнія» і за кілька хвилин окрема авіазагроза «на пуски у наш бік» в тому ж каналі). Мітигація — кілька слотів (2-3) на канал замість одного плюс LLM-tie-breaker для неоднозначних випадків, а не намагання зробити зіставлення повністю детермінованим.

## Альтернативи відхилені

- **Буквальний ТЗ Етап 1 (чистий RegEx без стану каналу)** — відхилено: структурний recall-провал (26-54% пропущених коротких повідомлень на ключових каналах) показаний на реальних даних, а не лише теоретично; додавання слів у словник цього не виправляє.
- **LLM на кожне повідомлення одразу, без рівнів 1-2** — відхилено: дорожче й повільніше без потреби (рівні 1-2 вже дають високий recall на цьому вузькому, повторюваному домені), і головне — саме в пікові моменти масованої атаки, коли LLM найпотрібніша, ризик rate limit найвищий (ТЗ п.16); критичний шлях виявлення не повинен залежати від зовнішнього API.
- **Покладатись на Telegram `reply_to_message_id` для звʼязку продовжень** — відхилено: перевірено напряму, що канали живого спостереження (RDS-prostir) взагалі не використовують reply-threading для власних продовжень.
