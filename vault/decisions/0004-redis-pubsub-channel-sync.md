# ADR-0004: Синхронізація списку моніторинг-каналів між адмін-панеллю і юзерботом

**Контекст:** список каналів керується через адмін-панель (`monitoring_channels` у PostgreSQL), а слухає їх окремий процес — юзербот (Telethon). Потрібен спосіб донести зміну списку до юзербота без рестарту процесу (сесія юзербота має жити ≥30 днів без втручання, ADR не пов'язаний з рестартами).

**Розглянуті варіанти:**
- Restart юзербота на кожну зміну — просто, але рве сесію
- Тільки polling БД (кожні 30-60с) — просто, але затримка до хвилини на нові канали
- Тільки Redis pub/sub — миттєво, але pub/sub у Redis не гарантує доставку (повідомлення губиться, якщо підписник в момент публікації тимчасово відвалився/перезапускається)

**Рішення:** обидва механізми одночасно.
- Redis pub/sub канал `channels:updates` — адмін-панель публікує подію при add/deactivate, юзербот підписаний і оновлює in-memory set `active_ids` миттєво
- Паралельно — periodic poll кожні 60 сек (`periodic_join_sync`) як страхувальна сітка на випадок втраченого pub/sub-повідомлення

Redis вже присутній в архітектурі як брокер (черга RegEx→LLM, дедуп-ключі з ТЗ п.9) — додатковий pub/sub-канал у тій самій інстанції не додає інфраструктурної складності.

**Технічна деталь реалізації:** `active_ids` — мутабельний `set[int]` у пам'яті юзербот-процесу, на який `NewMessage`-хендлер посилається через замикання (`func=lambda e: e.chat_id in active_ids`). Оновлення множини в місці (`clear()` + `update()`) не вимагає перереєстрації хендлера в Telethon.

**Приєднання до нового каналу:** сам запис у БД не дає юзерботу отримувати події — акаунт має бути підписаний (`JoinChannelRequest` для публічних, `ImportChatInviteRequest` для приватних за invite-хешем). Приєднання відбувається чергою з затримкою (`JOIN_DELAY_SECONDS = 15с`) між викликами, з `except FloodWaitError: sleep(e.seconds)` — масове швидке приєднання до багатьох каналів підвищує ризик бану того самого акаунта, який і так лише один без резерву в активному стані (див. п.3/п.12 ТЗ).

## Уточнення 2026-07-24: транзитний обрив Redis валив увесь юзербот-процес

Знайдено наживо (не тестом): `redis.exceptions.ConnectionError` ("The semaphore timeout period has expired" — Windows-специфічний минущий мережевий збій сокета, не справжня недоступність Redis) вискочив усередині `channel_updates_listener()`'s `async for message in pubsub.listen()`. `main()` запускає цю функцію через `asyncio.gather()` разом з `client.run_until_disconnected()` — необроблений виняток в ОДНІЙ з gather-задач скасовує решту й валить весь процес. Юзербот повністю зупинив моніторинг (жодне повідомлення не оброблялось) на ~15 хв, поки не був перезапущений вручну — той самий backlog-пункт "юзербот немає process-supervisor/авто-рестарту" (сесія 07-22) раніше вважався некритичним ("ручна зупинка під час паузи"), тепер стався як справжній необроблений краш під час активної роботи.

**Виправлено:** `channel_updates_listener()` обгорнуто в `while True` + `except redis.exceptions.ConnectionError` — при обриві логує (`logger.exception`), чекає 5с, перепідписується й продовжує. Пропуск одного pub/sub-сигналу під час перепідключення не критичний — `periodic_join_sync()` (60с страхувальна сітка, розділ "Рішення" вище) все одно підхопить будь-яку пропущену зміну списку каналів. `client.run_until_disconnected()` (сам Telethon-цикл обробки повідомлень) цей фікс не торкається — Telethon має власний `auto_reconnect` (ТЗ §12), тут ішлося конкретно про НАШ pub/sub listener без жодного захисту.

**Перевірено ізольовано:** підмінений Redis-клієнт, `pubsub.listen()` кидає `ConnectionError` на першій ітерації — функція не пропускає виняток назовні, логує, засинає (замоканий `asyncio.sleep`), перепідписується і на другій ітерації коректно обробляє повідомлення. Перезапущено юзербот наживо, 0 помилок.
