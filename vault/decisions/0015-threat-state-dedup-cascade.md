# ADR-0015: Вкладка "Сповіщення" — дедуплікація + каскадна стейт-машина

## Контекст

Три рівні конвеєра виявлення (ADR-0012) стабільні, але це шар виявлення, не сам продукт. Пряма перевірка коду (`grep -rn "send_message\|threat_state\|dedup_hash" app/`) підтвердила: жодного рядка, що надсилає сповіщення користувачам, `dedup_hash`/`confirmation_count` у `events_log` (ТЗ §14) визначені в схемі, але ніде не читаються й не пишуться, таблиць `threat_state`/`manual_overrides`/`trusted_users` (ТЗ §10/§11) не існувало.

Користувач попросив: (1) реальну розсилку (Telegram/WhatsApp) не робити зараз — окремий крок пізніше; (2) зробити вкладку "Сповіщення" в адмін-панелі, куди потрапляють вже проаналізовані, дедупльовані цілі — імітація того, що вийшло б на виході конвеєра. Уточнено прямими запитаннями (AskUserQuestion): дедуплікація потрібна (кілька каналів про одну ціль → один запис), каскад/автознижeння рівня з часом — теж потрібен, LLM на виході — спроектувати конвеєр із чітким швом, але сам виклик — стаб (бюджетне рішення ADR-0012 не змінюється), роль LLM (коли стане реальним) — і компонує текст сповіщення, і розв'язує неоднозначні випадки (`channel_state_ambiguous`).

## Рішення

**`threat_state` — кілька одночасних відкритих рядків за локацією, не один глобальний рядок і не один на канал.** Той самий принцип, що вже двічі підтверджений у цьому проєкті: Рівень 2 (`channel_state.py`) довелось розширити з 1 до 3 слотів на канал (одночасні непов'язані цілі), `DecisionTrace.location` довелось зробити кортежем (маршрут через кілька топонімів). Один глобальний рядок злив би "КАБ по Сумах" і "шахед на Шостку" в один рівень — та сама втрата інформації. Один рядок на канал унеможливив би саму мету дедуплікації (різні канали мають підтверджувати ОДНУ ціль). Матчинг нового сигналу до відкритого рядка — перетин множин локацій, той самий принцип, що `channel_state._write_explicit` для вибору слота, лише міжканально.

**`threat_notifications` — окрема append-only таблиця**, не той самий рядок, що `threat_state`. Той самий принцип, що `events_log` для редагувань: `threat_state` — жива мутабельна ціль (мусить бути мутабельною, ТЗ §10 явно проти "стан лише в пам'яті процесу"), але стрічка "Сповіщення" має показувати ІСТОРІЮ переходів (нова/ескалація/зниження/відбій), не один рядок, що тихо перезаписується. Запис у `threat_notifications` — лише на РЕАЛЬНІ переходи; мовчазні підтвердження того самого рівня від додаткових каналів НЕ породжують запис (інакше дедуплікація не дала б жодного практичного зменшення шуму — саме той ефект, заради якого й затівалась).

**Дедуплікація (ТЗ §9) прямо годує каскад (ТЗ §10), не два окремі механізми:** "відбій" від ОДНОГО каналу не гасить ціль негайно — лише коли `status_confirming_channels` набирає 2+ різних джерела, запускається таймер T+1хв. Один голий "відбій" не губиться назавжди — 20-хвилинний фолбек "локаційно втрачена ціль" (`cascade_tick`) все одно рано чи пізно знизить рівень навіть без другого підтвердження.

**Каскадні таймінги (T+1хв/T+6хв/T+21хв) реалізовані як дельти між послідовними кроками, не абсолютні мітки від єдиного t0** — при переході на `orange` планується наступний крок за +5хв (разом T+6 від початкового +1хв), при переході на `yellow` — за +15хв (разом T+21). Це дає ту саму точну арифметику ТЗ §10 без додаткової колонки для запам'ятовування моменту першого підтвердження.

**LLM (Рівень 3) — стаб, не реальний виклик.** `app/common/llm.py` — окремий файл з двома функціями (`compose_alert_text`, `resolve_ambiguous_slot`), обидві повертають `is_stub=True`. Текст сповіщення складає детермінований шаблон (рівень+локація+evidence+лічильник підтверджень), без жодного API-виклику/ключа — бюджетне рішення ADR-0012 не переглядається цим ADR. `composed_by='template_stub'` у БД і окремий бейдж в UI ("📝 шаблон") — навмисно інший, ніж уже зарезервований `origin_map['llm'] = '🤖 ШІ'` у `_level_badge.html`, щоб стаб ніколи не сплутати з майбутньою реальною відповіддю при читанні БД/логів.

**Ambiguous tie-break у Рівні 2 переведено на виклик `llm.resolve_ambiguous_slot()`** замість inline `max(slots, key=...)` в `channel_state.resolve()` — поведінка не змінюється (та сама евристика "найновіший"), але з'являється реальна точка виклику під майбутню роль (б), не лише декларація в ADR-0012.

**Каскад тікає в адмін-процесі** (`asyncio.create_task` у `lifespan()`, той самий патерн, що `periodic_join_sync` у юзерботі, лише в іншому процесі) — свідомо без третього непіднаглядного процесу (уже є backlog-пункт про відсутність supervisor). Компроміс: якщо адмін-панель не працює, каскад не тікає, але нічого не втрачається (`downgrade_scheduled_at` живе в Postgres, перший тік після рестарту наздоганяє прострочене) — затримка, не втрата даних.

## Знайдений і виправлений по дорозі баг

Власний ізольований тест (`record_signal()` напряму, без очікування живого трафіку — канали були тихі) одразу впіймав реальний баг: текст сповіщення про ЕСКАЛАЦІЮ показував `threat_type_evidence` СТАРОГО рівня (напр. "бпла" при ескалації на red через "ракет"), бо `_write_notification` читав evidence зі стейл-знімка рядка (`row`, зафіксованого ДО `UPDATE`), а не з нового `trace`. Виправлено — `threat_type_evidence` тепер явний параметр функції, переданий окремо в кожній точці виклику, не читання з можливо застарілого рядка. Той самий клас перевірки, що вже раз впіймав баг у `channel_state.resolve()` до підключення наживо (ADR-0012, "Уточнення 2026-07-21").

Окремо — сам тестовий скрипт лишив по собі синтетичний рядок у `events_log` (`source_channel='-100test'`, нечислове значення) і зламав `/events` (500, `bigint`-каст у JOIN очікує лише числові `source_channel`) — знайдено одразу через фоновий `Monitor` на логи, прибрано, тест переписано з валідним числовим ідентифікатором.

## Альтернативи відхилені

- **Один глобальний рядок `threat_state`** (буквально як у ТЗ §10 псевдокоді, де `cascade_tick(state)` виглядає як один стан) — відхилено: злив би різні одночасні локації в один рівень, той самий клас втрати інформації, що вже двічі виправлявся в цьому проєкті іншими механізмами.
- **Третій окремий процес під каскадний тік** — відхилено: додав би ще одну непіднаглядну точку відмови (backlog уже фіксує відсутність supervisor навіть для двох існуючих процесів), а функціонально різниця мінімальна — Postgres-стан переживає рестарт будь-якого процесу.
- **Реальний виклик LLM зараз** — відхилено користувачем прямо в AskUserQuestion: бюджетне рішення (ADR-0012) не приймається цим кроком, лише конвеєр спроектований під легке підключення пізніше.
