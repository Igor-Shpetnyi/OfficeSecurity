# ADR-0011: Заглушки вкладень замість голого "—"

## Контекст

Користувач помітив підозру: якщо в Telegram-повідомленні є і фото, і текст, у панелі показувалось лише "—", текст губився. Перевірка напряму через Telegram API (`client.get_messages`) виявила справжню причину: **альбоми (`grouped_id`)**. Коли канал публікує кілька фото одним постом з одним підписом, Telegram зберігає підпис лише в ОДНОМУ повідомленні групи — решта фото приходять окремими повідомленнями з порожнім текстом. Користувач бачить це в Telegram як один пост; наш юзербот отримує кожне фото окремою подією. Це не баг захоплення тексту — `raw_text` порожній по-справжньому для цих повідомлень.

Окремо виявлено (те саме розслідування): деякі повідомлення отримують медіа не одразу, а через `edit` (текст спершу без фото, потім відредаговано з доданим фото) — `events_log` це коректно відображає різними `media_type` для 'new' і 'edit' рядків одного повідомлення.

## Рішення

За проханням користувача — замість голого "—" показувати заглушку вкладення (іконка + підпис: "Фото", "Відео" тощо), з явним наміром пізніше підставити самі оригінальні файли замість заглушок.

**Структурований тип, не рядок для показу:** `events_log.media_type` зберігає короткий код (`photo`/`video`/`gif`/`sticker`/`voice`/`audio`/`poll`/`document`/`other`), НЕ готовий текст на кшталт "📷 фото". Людський підпис і іконка рахуються з коду в `app/common/media.py` (мапа `MEDIA_LABELS`) і `app/admin/templates/_media_icon.html` (Jinja-макрос з SVG на тип). Це замінило попередню чернетку (2026-07-21, той самий день) — там `media_label` зберігав готовий рядок з емодзі напряму в БД; довелось переробити, щойно стало зрозуміло, що це не остаточна форма — заглушки мають бути замінні на реальний рендер вкладення, а рядок-з-емодзі для цього не підходить: код типу дає шаблону змогу пізніше сказати "якщо photo і є завантажений файл — `<img>`, інакше — іконка-заглушка" без розбору тексту.

**`grouped_id` зберігається про запас**, хоч і не використовується зараз — природна точка розширення під майбутнє об'єднання альбому в одну картку (кілька фото + спільний підпис як один UI-елемент, а не N карток). Не реалізовувати зараз без окремого запиту користувача — сама механіка об'єднання (яку картку показувати "головною", як рендерити decorative "+3 фото" тощо) це окрема фіча.

**Заглушка показується незалежно від наявності тексту**, не лише як фолбек на порожній текст — саме це виправляє первинну скаргу: пост "фото + підпис" тепер показує і іконку-заглушку, і повний текст підпису одночасно, а не одне замість іншого.

## Виявлений і виправлений по дорозі баг

Перша версія показувала заглушку на дашборді (`_dashboard_content.html`) лише коли `raw_text` порожній (`{% elif %}` після перевірки тексту) — для повідомлень з фото ТА текстом заглушка на дашборді просто не з'являлась (текст показувався сам, без індикації вкладення), хоча на основній сторінці "Події" (`_events_content.html`) заглушка й текст уже показувались разом. Знайдено власною перевіркою (звірка кількості `attachment-stub` в HTML з прямим SQL-запитом того самого набору подій), не користувачем. Виправлено для консистентності між сторінками.

## Альтернативи відхилені

- **Одразу зберігати готовий текст-лейбл з емодзі в БД** — перша чернетка цього ж дня; відхилено одразу, як тільки стало зрозуміло про план на реальні вкладення в майбутньому — типізований код розширюється, готовий рядок — ні.
- **Одразу реалізувати об'єднання альбомів в одну картку** — це окрема, більша фіча (яка картка "головна", як рендерити рахунок фото в групі), користувач просив саме заглушки; `grouped_id` зберігається, щоб не переробляти схему, коли до цього дійде.
