# ADR-0009: Колір ідентифікації каналу — hue з аватарки, фіксовані lightness/saturation

## Контекст

Канали в стрічці подій і списках потрібно візуально розрізняти (схожі імена: `@sumygo`/`@sumyliketop`/`@sumitruexa` зливались при однаковому кольорі). Спочатку — хеш назви каналу % 4, що мапився на 4 фіксовані токени дизайн-системи (`--positive`/`--warning`/`--badge-pro`/`--negative`). За проханням користувача — колір має бути "реальним" з аватарки каналу, не хешем.

## Чому саме так

Перша реалізація — простий **середній колір усього зображення** (`Image.resize((1,1))`). Виявилась проблема при перевірці: середнє часто виходить темним/приглушеним (лого з темним фоном, або змішування багатьох кольорів у "багнюку"), що на темній темі давало контраст тексту до фону картки **~1.2:1** (виміряно точно через relative luminance formula, не на око) — практично нечитабельно. WCAG AA вимагає мінімум 4.5:1.

Рішення: з аватарки береться **лише hue** (тон) середнього кольору — це й є унікальна "айдентика" каналу. Lightness/saturation фіксуються (`L=0.55, S=0.55` — `app/common/avatars.py::_legible()`), той самий підхід і для реальних фото, і для процедурного фолбека каналів без фото (`fallback_color()`, hue з хеша назви). Контраст піднявся до 3.46-7.58:1 на реальних даних.

Технічно: колір зберігається в `monitoring_channels.avatar_color` (`#rrggbb`), рахується юзерботом одразу після завантаження фото (`fetch_avatar()` → `extract_avg_color()`), передається в шаблони через inline CSS custom property `--chan-color`, а CSS-правила (`app/admin/static/css/theme.css`) читають її через `var()` — жодних хардкод-кольорів per-канал у CSS, працює для будь-якого числа каналів (не обмежено 4 бакетами, як раніше).

## Альтернативи відхилені

- **Простий average color без нормалізації** — відхилено через контраст (див. вище).
- **Dominant color через quantize/палітру** (замість average) — точніше відображав би реальний "головний" колір лого, але складніше й повільніше; для 38×38px акценту в UI різниця непринципова, average+hue-normalize достатньо.
- **4 фіксовані токени дизайн-системи (хеш)** — попередня версія; відхилено за прямим запитом користувача, і об'єктивно гірше розрізняла схожі назви при рості кількості каналів понад 4.

## Уточнення 2026-07-21: near-gray-average edge case

Виявлено на реальних даних (канал "Повітряні Сили ЗС України" — темно-синій фон + жовтогарячий тризуб): контрастні насичені зони в одному лого гасять одна одну в наївному RGB-усередненні, даючи майже ахроматичний сірий піксель (`sat≈0.03`). Hue такого пікселя чисельно нестабільний (шум у кілька одиниць RGB довільно перевертає значення), а `_legible()` все одно перетворює його на "впевнений" насичений колір, що не має нічого спільного з реальним зображенням — на 10 реальних каналах зачепило 4 (`sat < 0.15`). Заразом виявлено, що `resize((1,1))` без явного `resample` використовував дефолтний bicubic, не справжнє усереднення.

Виправлено без відходу від базового рішення (average+hue-normalize лишається): `Image.Resampling.BOX` для коректного усереднення, і якщо `sat < 0.15` — `extract_avg_color()` повертає `None`, викликач падає на вже наявний `fallback_color()`. Це не спростовує рішення "не ускладнювати до quantize/dominant-color" з розділу вище — просто патчить конкретний нестабільний edge-case дешевшим запобіжником, ніж повноцінний dominant-color алгоритм.

**Питання від користувача:** чому фолбек — хеш назви, а не якийсь колір все-таки "з палітри" аватарки (напр. найнасиченіший піксель зображення, без повної квантизації)? Запропоновано як легший проміжний варіант — **користувач свідомо відхилив**, лишити поточний хеш-фолбек як є. Не re-litigate цю ідею повторно без нового приводу.
