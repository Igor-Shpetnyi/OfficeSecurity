# Технічне завдання: Telegram-система ситуаційної обізнаності «Безпека Офісу — Суми»

**Версія:** 2.0 (розширена)
**Статус:** Робочий документ для самостійної розробки

---

## 1. Мета та критерії успіху

**Мета:** автоматичне визначення рівня загрози для офісу в м. Суми на основі моніторингу Telegram-каналів та сповіщення співробітників з затримкою не більше 1-2 сек для сирого сигналу.

**Критерії готовності MVP:**
- Бот стабільно тримає сесію юзербота ≥ 30 днів без ручного втручання
- Затримка RegEx-сигналу ≤ 2 сек від публікації в моніторинг-каналі
- ШІ-уточнення приходить окремим повідомленням протягом 1-3 сек після сирого сигналу
- Дедуплікація коректно гасить дублі з 3+ каналів по одній події
- Каскадне зниження рівня спрацьовує без ручного втручання і коректно переривається новим тригером

---

## 2. Архітектура

```
Telegram-канали (моніторинг)
        │
        ▼
[Userbot: Telethon] ── event handler (NewMessage)
        │
        ├──► RegEx/keyword engine (0-100мс) ──► сирий сигнал ──► Notification Service ──► Telegram-бот → офісний чат
        │
        └──► Черга (Redis) ──► LLM Worker (async) ──► уточнення/корекція рівня ──► Notification Service
                                                              │
                                                              ▼
                                                    PostgreSQL (стан, лог, RLHF-датасет)
                                                              │
                                                              ▼
                                                    State Machine (таймери каскаду, TTL)
                                                              │
                                                              ▼
                                                    Admin Panel (FastAPI) ──► ручне керування
```

**Ключовий принцип:** RegEx-двигун і LLM працюють **паралельно, не послідовно**. RegEx ніколи не чекає на LLM — це та сама причина, чому "гібридна модель" з п.1 оригінального ТЗ не суперечить вимозі 1-2 сек, а доповнює її.

## 3. Інфраструктура та хостинг

### Порівняння варіантів

| Варіант | Плюси | Мінуси |
|---|---|---|
| Локальна машина в офісі | Нуль щомісячних витрат, повний фізичний контроль | **Критично:** під час обстрілу саме тоді, коли система найпотрібніша, може зникнути світло/інтернет в офісі. Система відмовить в момент істини |
| Хмара (Hetzner/DigitalOcean, EU) | Стабільне живлення й мережа поза зоною ризику, DDoS-захист на рівні провайдера, снапшоти для бекапу | Щомісячна оплата (невелика — CX22/CX32 клас вистачить) |
| Великі хмари (AWS/GCP) | Надійність, масштабованість | Надлишково дорого і складно для цієї задачі, зайва мережева латентність через регіони |

**Рекомендація: Hetzner Cloud (CX22, Нюрнберг або Фінляндія)**, ~4-6 €/міс. Обґрунтування: система, ціль якої — попереджати про загрозу життю, не може залежати від того самого джерела нестабільності (відключення світла в Сумах), від якого вона захищає. Це не обговорюється — локальна машина тут відпадає одразу.

Одна VPS тягне все: юзербот + FastAPI + PostgreSQL + Redis. Розділяти на кілька серверів для MVP немає сенсу — зайва складність без вигоди на цьому масштабі трафіку.

### Резервування
- Другий обліковий запис-юзербот (інший номер) в резерві, сесія не активна, підключається вручну/скриптом при бані основного
- Щоденний бекап PostgreSQL (pg_dump) у зовнішнє сховище (наприклад, окремий S3-сумісний бакет, не на тому ж VPS)
- Моніторинг живості самого бота (healthcheck) через зовнішній сервіс (напр. UptimeRobat/healthchecks.io) — якщо бот сам впав, хтось має про це дізнатись, а не чекати мовчання

## 4. Технологічний стек

| Компонент | Вибір | Чому |
|---|---|---|
| Userbot | **Telethon** (не Pyrogram) | Кращий event-based API з коробки саме для моніторингу каналів, детальніша документація по FloodWait/reconnect |
| Мова | Python 3.12 | Весь екосистема (Telethon, LLM SDK, asyncio) нативно async, що критично для паралельного RegEx/LLM |
| Черга | Redis (Streams або списки + TTL-ключі) | TTL-ключі ідеально лягають на каскадні таймери з п.8; та сама Redis-інстанція під rate-limit семафор для LLM |
| БД | PostgreSQL | Стан системи, лог подій, RLHF-датасет ручних корекцій — реляційна модель тут природна |
| Admin panel | FastAPI + мінімальний фронт (HTMX або просто Jinja2-шаблони) | Не варто городити React заради однієї сторінки з кнопками рівня і формою причини |
| Процес-менеджер | systemd (не Docker для MVP) | Простіше дебажити reconnect-логіку юзербота наживо на одній VPS; Docker — окремий рефактор для Етапу 2 |

## 5. LLM-модуль: порівняння варіантів

| Варіант | Затримка | Вартість (орієнтовно, 500 повід./день) | Складність | Ризик у пікові дні |
|---|---|---|---|---|
| Хмарний API (GPT-4o-mini / Claude Haiku) | 0.5-2 сек | $3-8/міс при короткому JSON-виводі | Мінімальна — просто SDK-виклик | Rate limit при сплеску трафіку (масована атака = шквал повідомлень з каналів) |
| Локальна LLM (напр. Llama 3.1 8B / Qwen 2.5 7B, квантизована) | 1-4 сек на CPU, <1 сек на GPU | $0 за токени, але потрібен сервер з GPU (~30-60 €/міс оренда) або CPU-інференс (повільніше, якість гірша на класифікації складних векторів) | Висока — розгортання, квантизація, підтримка моделі | Немає зовнішньої залежності, але власний ресурс може захлинутись при сплеску так само |
| Гібрид: хмарний API як основний + fallback на чисті RegEx-правила при недоступності API | 0.5-2 сек (норма), 0мс (fallback) | Як хмарний варіант | Середня — треба queue + circuit breaker | Найнижчий — система ніколи не блокується повністю |

**Рекомендація: гібрид з хмарним API як основним.**

Обґрунтування: локальна LLM для цієї задачі — це складність заради складності. Точність класифікації складних векторів ("транзит повз" vs "курс на Суми") у 7-8B моделей помітно нижча за GPT-4o-mini/Claude Haiku, а ціна питання — кілька доларів на місяць. Головний ризик хмарного API — rate limit саме в момент масованої атаки — закривається fallback-логікою (п.9), а не відмовою від хмари. Тримати GPU-сервер заради економії $5/міс — погана угода часу розробника.

Якщо бюджет == 0 категорично, тоді локальна LLM виправдана, але закладай +2-3 тижні на саму інтеграцію та тюнінг проти хмарного варіанту (1-2 дні).

## 8. Географічний модуль

### MVP (Етап 1)
Статичні списки топонімів по зонах + матчинг тексту повідомлення проти списку. Просто, швидко в розробці, достатньо точно для 90% кейсів — назва населеного пункту зазвичай явно фігурує в тексті моніторинг-каналів.

### Масштабування (Етап 2)
- База координат (GeoJSON) для кожного топоніма з розділу 4.1/4.2 вихідного ТЗ
- Формула Гаверсинуса для розрахунку відстані від точки виявлення до офісу
- Розрахунок часу підльоту = відстань / середня швидкість типу загрози (з таблиці розділу 6)
- Векторна логіка: якщо точка виявлення лежить на одному з 5 векторів (розділ 4.2 вихідного ТЗ) і рухається у бік Сум — ескалація; якщо вектор веде в інший бік — це "транзит повз", рівень не підвищується

**Важливо:** Етап 2 не варто починати, поки MVP на статичних списках не пропрацював мінімум місяць у бою — реальні дані з каналів покажуть, які топоніми/формулювання система пропускає, і GeoJSON-шар варто будувати вже під ці прогалини, а не наосліп.

## 9. Словник тригерів, анти-спам та дедуплікація

Словник тригерів (🟢/🟡/🟠/🔴) — без змін від вихідного ТЗ, розділ 5.

### Механізм дедуплікації (конкретизація п. "Anti-Spam" з оригіналу)

1. Нормалізація тексту повідомлення (lowercase, прибрати пунктуацію/емодзі/посилання)
2. Хеш нормалізованого тексту (SHA-256 достатньо) + `threat_type` + `target_location` з відповіді ШІ
3. Redis-ключ `dedup:{hash}` з TTL 30 сек — якщо ключ вже існує, повідомлення вважається дублем, лічильник підтверджень +1, нове сповіщення не шлеться
4. Якщо той самий `threat_type` + `target_location`, але хеш тексту інший (перефразовано в іншому каналі) — семантичне зіставлення через ту саму LLM-класифікацію (поле `target_location`+`threat_type` з JSON-відповіді і є точкою зіставлення, а не сирий текст)
5. Лічильник підтверджень з різних каналів (2+ джерела) — саме він є тригером для правила "чисто"/"відбій" з п.8 вихідного ТЗ

Це розв'язує обидва сценарії з попереднього аналізу: "один шахед з 5 каналів" (дедуп по hash+location) і "5 різних шахедів одночасно" (різні location → різні події → окремі сповіщення).

## 10. Каскадний алгоритм зниження (Downgrade Logic) — деталізація

Стейт-машина зі станом у PostgreSQL (не тільки в пам'яті процесу — інакше рестарт бота скидає всі таймери й рівень назад у 🟢, що небезпечно).

**Таблиця `threat_state`:**
```sql
CREATE TABLE threat_state (
    id SERIAL PRIMARY KEY,
    current_level VARCHAR(10) NOT NULL DEFAULT 'green', -- green/yellow/orange/red
    location VARCHAR(100),
    threat_type VARCHAR(50),
    triggered_at TIMESTAMPTZ,
    downgrade_scheduled_at TIMESTAMPTZ, -- коли має спрацювати наступна фаза каскаду
    last_update_source VARCHAR(50), -- 'auto' / 'manual:<user_id>'
    is_locationally_lost BOOLEAN DEFAULT FALSE,
    lost_since TIMESTAMPTZ
);
```

**Логіка (псевдокод воркера, що тікає раз на 5-10 сек):**
```python
def cascade_tick(state):
    now = utcnow()

    # Абсолютне правило переривання
    if new_trigger_arrived and new_trigger.level > state.current_level:
        set_level(new_trigger.level, reset_timers=True)
        return

    if state.current_level == 'red' and neutralization_confirmed:
        schedule_downgrade('orange', now + timedelta(minutes=1))

    if state.downgrade_scheduled_at and now >= state.downgrade_scheduled_at:
        next_level = CASCADE_MAP[state.current_level]  # red->orange->yellow->green
        set_level(next_level)
        if next_level != 'green':
            schedule_next_cascade_step(next_level, now)

    # Timeout для "локаційно втрачена ціль"
    if state.is_locationally_lost and now - state.lost_since >= timedelta(minutes=20):
        downgrade_one_step(state)
```

Таймінги — точно як у вихідному ТЗ (п.8): T+1хв → 🟠, T+6хв → 🟡, T+21хв → 🟢. Кожен крок каскаду — окремий запис у лог (для RLHF та для розбору інцидентів пізніше).

## 11. Адмін-пульт

**Функціонал:**
- Поточний стан (рівень, локація, з якого джерела) на одному екрані
- Кнопка ручної зміни рівня — **обов'язкове** текстове поле причини (без причини кнопка неактивна), запис у `manual_overrides` з `user_id`, `timestamp`, `reason`, `old_level`, `new_level`
- Перегляд логу подій за останні 24 год
- Керування списком довірених користувачів (хто має право на `/buh`, `/uav` команди)
- Керування списком моніторинг-каналів (додати/прибрати)

**Автентифікація:** Telegram Login Widget (прив'язка до Telegram-акаунту адміна) — не варто городити окрему систему логінів/паролів заради панелі, яку відкриває 1-2 людини. Або простіше — Basic Auth за VPN/whitelist по IP, якщо адмінів реально один-два і вони завжди з відомих IP.

## 12. Відмовостійкість юзербота

- `telethon.errors.FloodWaitError` — обов'язковий except з `asyncio.sleep(e.seconds)`, інакше акаунт заблокує ще жорсткіше
- Reconnect: Telethon має вбудований `auto_reconnect=True`, але додатково — health-check event кожні 60 сек (читання власного статусу), якщо немає — примусовий `client.disconnect()` + `client.connect()` з backoff (1с, 2с, 4с... до 60с)
- Сесія юзербота — StringSession у зашифрованому вигляді в `.env`/secret manager, **не** файл `.session` у git
- Фіксований IP і незмінний device fingerprint (Telethon `device_model`, `system_version`) — Telegram банить швидше за "підозрілі" зміни

---

## 13. Черга подій та graceful degradation

- Redis Streams як черга: `regex_engine` пише event → `llm_worker` читає, класифікує, пише назад
- Rate-limit семафор: якщо LLM API повертає 429 або чергу не встигає розбирати (довжина черги > N), воркер перемикається в режим "тільки RegEx" — сирі сигнали й далі йдуть, просто без ШІ-уточнення/фільтрації шуму, з міткою "⚠️ без ШІ-верифікації" в тексті сповіщення
- Circuit breaker: 3 поспіль невдалі виклики LLM API → пауза 30 сек перед наступною спробою, щоб не добивати API, що й так лежить

## 14. Схема БД (додатково до `threat_state`)

```sql
CREATE TABLE events_log (
    id SERIAL PRIMARY KEY,
    raw_text TEXT,
    source_channel VARCHAR(100),
    detected_at TIMESTAMPTZ,
    regex_matched_level VARCHAR(10),
    llm_response JSONB, -- сира відповідь ШІ за схемою з п.6 оригінального ТЗ
    dedup_hash VARCHAR(64),
    confirmation_count INT DEFAULT 1
);

CREATE TABLE manual_overrides (
    id SERIAL PRIMARY KEY,
    user_id BIGINT,
    old_level VARCHAR(10),
    new_level VARCHAR(10),
    reason TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE trusted_users (
    user_id BIGINT PRIMARY KEY,
    telegram_username VARCHAR(100),
    can_trigger_red BOOLEAN DEFAULT TRUE,
    added_by BIGINT,
    added_at TIMESTAMPTZ DEFAULT now()
);
```

## 16. Ризики

| Ризик | Мінімізація |
|---|---|
| Бан юзербот-акаунта | Резервний акаунт напоготові (п.3, п.12) |
| Rate limit LLM API під час масованої атаки | Fallback на чистий RegEx (п.13) |
| Хибне спрацювання 🔴 через жарт у чаті довірених | ШІ-фільтр контексту перед прийняттям `/buh`/`/uav` (п.7 оригінального ТЗ) |

| Пропуск нового топоніма/сленгу | Лог усіх `is_relevant=false` для періодичного ручного review — можливо, ШІ щось відфільтрував помилково |
